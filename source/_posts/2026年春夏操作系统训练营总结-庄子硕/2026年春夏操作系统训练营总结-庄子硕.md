## 项目概述

本项目是 2026 春夏开源操作系统训练营（Async 专题）的结题总结，系统性地探索了从用户态到内核态的异步并发机制。

整个项目分三个阶段递进：

1. **Task 1 — 用户态并发模型性能对比**：实现基于进程(fork)、线程(std::thread)、协程(tokio async)三种模型的网络爬虫，在 33 所高校首页上对比延时分布、吞吐率和内存开销。
2. **Task 2 — 用户态调度器设计**：实现 CFQ (Completely Fair Queuing) 公平调度器，并适配到四种执行流模型（OS 线程、回调、有栈协程、无栈协程），手写 Waker、Reactor、绿色线程上下文切换。
3. **Task 3 — 内核态协程探索**：研究 Embassy 嵌入式异步运行时，为 embassy-preempt 扩展同优先级 Round-Robin 调度和 Mutex（优先级天花板协议）。

项目仓库：[OS_2026_Async_camp](https://github.com/Gingkoleaves/OS_2026_Async_camp)

## 理论学习收获

训练营期间，我结合 Rust Async Book、200 行绿色线程/无栈协程参考实现、Tokio 源码分析等材料，系统学习了以下核心概念。

**并发 ≠ 并行 ≠ 异步**。并发是"能处理多个任务"，并行是"同时处理多个任务"，而异步是一种完全不同的编程模型——它让你在不阻塞线程的前提下等待外部事件。Rust 选择了基于 poll 的 Future 而非有栈协程，核心动机是"零成本抽象"：async 函数编译为状态机（单一 enum），poll 走静态分发，drop 即取消——没有任何额外运行时开销。

**Waker 是连接 Reactor 和 Executor 的桥梁**。当 Future 返回 Pending，它通过 Context 注册 Waker；当 I/O 就绪或定时器触发，Reactor 调用 `waker.wake()`，Executor 收到通知后重新 poll。Waker 本质是两个 8 字节指针（data + vtable），是手搓的 trait object——这是 Rust 异步运行时最精妙的设计之一。

**协作式调度的基本局限**。绿色线程和无栈协程都是协作式模型——任务必须主动 yield 才能让出 CPU。这意味着一个不 yield 的 CPU 密集任务会饿死所有同优先级任务。OS 线程的抢占式调度解决了这个问题，但引入了上下文切换开销。embassy-preempt 用中断实现了嵌入式场景下的抢占，是两者的巧妙结合。

**优先级调度生效的前提是 yield 注入**。没有 yield 点时，协作式调度器的优先级形同虚设。本文实验中每 100 次迭代注入一个 yield 点，产生约 4-6ms 的有效时间片，优先级差异才得以体现。

**外置调度层的瓶颈在分发不在选择**。BinaryHeap 的 O(log n) 选择不是瓶颈（33 次仅 50ns），真正拖慢性能的是串行分发循环（33 次 spawn）。这教会我诊断性能问题时 user time vs wall time 是关键线索：user time 相同而 wall time 不同，问题一定在调度/分发层。

## 一、网络爬虫性能对比 (Task 1)

### 实验设计

爬取 33 所中国高校首页的纯文本内容，在三种并发模型下分别以 4、32、64 并发度运行，测量总耗时、吞吐率、延迟分布（P50/P95/Min/Max）和峰值内存。

### 4 并发结果

| Metric | async (tokio) | process (fork) | thread (std) |
|--------|:---:|:---:|:---:|
| Total time | **3.22 s** | 8.79 s | 6.38 s |
| Throughput (KB/s) | **988.5** | 367.7 | 498.5 |
| Peak RSS (MB) | 17.9 | **13.8** | 18.5 |
| Avg latency (ms) | 953 | **523** | 451 |
| P95 latency (ms) | 2262 | 1654 | **1068** |

> **分析**：低并发时 async 的 P95 延迟最高（2262ms），少数任务被饥饿等待。process 延迟最稳定但吞吐最低——fork+exec 开销约占总时间 30%。

### 32 并发结果

| Metric | async (tokio) | process (fork) | thread (std) |
|--------|:---:|:---:|:---:|
| Total time | **2.03 s** | 3.09 s | 2.92 s |
| Throughput (KB/s) | **1565.4** | 1046.0 | 1088.1 |
| Peak RSS (MB) | 21.4 | **16.1** | 30.2 |

> **分析**：async 全面领先——epoll 多路复用 + 用户态调度避免了线程上下文切换开销。thread 内存达 30.2MB（是 process 的 1.9 倍）。

### 64 并发结果

| Metric | async (tokio) | process (fork) | thread (std) |
|--------|:---:|:---:|:---:|
| Total time | 1.76 s | **1.72 s** | **1.62 s** |
| Throughput (KB/s) | 1804.3 | 1875.3 | **1963.1** |

> **分析**：64 并发下三者差距缩小。process 和 thread 受益于 OS 级并行调度，与 async 的用户态调度性能接近。

### 小结

- **低并发 (≤4)**：async > thread > process
- **中并发 (32)**：async 明显最优，epoll 优势展现
- **高并发 (64)**：三者接近，OS 级并行优势显现
- **内存**：process 最省（独立地址空间），thread 最高（每线程 8MB 栈）
- **延迟稳定性**：thread 的 P95 最低且最稳定

## 二、CFQ 公平调度器设计 (Task 2)

### 设计原理

CFQ (Completely Fair Queuing) 调度器的核心是一个 `BinaryHeap`，按虚拟运行时间 (vruntime) 排序：

$$\text{vruntime} \mathrel{+}= \frac{\text{time\_slice} \times 1024}{\text{weight}}$$

- **权重表**：参考 Linux `prio_to_weight`，40 级优先级，相邻比 ≈ 1.25x
- **新任务**：初始 vruntime = 当前 min_vruntime（防止饿死）
- **tie-breaking**：相同 vruntime 时先按权重排序（高权重优先），再按 FIFO 序号

```
优先级 0 (最高) → weight = 88761
优先级 20 (默认) → weight = 1024
优先级 39 (最低) → weight = 15
```

### 四种执行流模型的 CFQ 适配

| 模型 | 实现 | 调度方式 | 代码量 |
|------|------|---------|:---:|
| **OS 线程** | `CfqThreadPool` | 元数据追踪 vruntime | ~200 行 |
| **回调模型** | `CfqScheduler<Box<dyn FnOnce()>>` | 直接入堆调度 | ~50 行 |
| **有栈协程** | Green Thread Runtime | `t_yield()` 时 CFQ 选择 | ~360 行 |
| **无栈协程** | `CfqExecutor` | poll 后按耗时更新 vruntime | ~400 行 |

### 公平性验证（58 个测试全部通过）

| 测试场景 | 预期 | 实测结果 |
|---------|------|---------|
| 等权重 3 任务公平轮转 | 各 ~100 次调度 | 90-110（误差 < 15%） |
| 权重比 ~3:1 (prio 0 vs 5) | ratio > 2.0 | ratio ≈ 3.05 |
| 10 高权重 + 1 最低权重 | 低优先级不饿死 | 至少被调度 1 次 |
| 新任务 min_vruntime 初始值 | 不被旧任务惩罚 | 新任务先执行 |
| 接近优先级交错 (19/20/21) | 有序但非独占 | 交错调度确认 |
| 相邻权重比 ~1.25 | [1.2, 1.3] | 1.25 ✓ |

### 技术亮点

**手写 Waker（虚表机制）**：

```rust
// Waker = data指针 + vtable指针（各8字节），本质是手搓的 trait object

    clone_fn,   // Arc::clone
    wake_fn,    // thread::unpark
    wake_by_ref_fn,

```

**有栈协程上下文切换（x86_64 naked_asm）**：

```asm

; 保存当前寄存器到 old (rdi)，从 new (rsi) 恢复

; ... 7 个 callee-saved 寄存器
```

## 三、CPU 密集型任务多运行时对比

### 实验设计

设计了 4 种 CPU 密集型任务，覆盖不同计算模式，在 6 种运行时上以不同并发度运行：

| 任务 | 计算特征 | Light | Medium | Heavy |
|------|---------|-------|--------|-------|
| **校验和** | 内存带宽敏感 | 1MB×10轮 | 4MB×50轮 | 16MB×100轮 |
| **质数筛选** | 整数运算密集 | limit=100K | limit=1M | limit=5M |
| **矩阵乘法** | 浮点运算密集 | 64×64 | 256×256 | 512×512 |
| **哈希计算** | 混合运算 | 64KB×50轮 | 256KB×200轮 | 1MB×500轮 |

### 核心发现

concurrency=4，checksum-light (1MB×10轮校验和) 的实测数据（本机：24 核，Linux 6.8）：

| Runner | 总耗时 | 吞吐率 | 每任务延迟 | RSS | 分析 |
|--------|:---:|:---:|:---:|:---:|------|
| **thread (std)** | **73ms** | **54.5 t/s** | 70.7ms | 7.8MB | OS 抢占式并行 |
| tokio (spawn_blocking) | 76ms | 52.1 t/s | 73.4ms | 4.0MB | 多核 work-stealing |
| process (fork) | 74ms | 53.4 t/s | 72.0ms | 7.9MB | 独立地址空间 |
| green-thread (CFQ) | 286ms | 14.0 t/s | 71.6ms | 8.9MB | 单线程协作式 |
| stackless (CFQ) | 277ms | 14.4 t/s | 69.5ms | 8.9MB | 单线程 Future |
| embassy (sim) | 277ms | 14.4 t/s | 69.3ms | 8.9MB | 单线程优先级 |

**关键结论**：
1. **每任务延迟在所有模型间几乎相同**（69.3-73.4ms）——差异在测量噪音范围内
2. 总耗时差异完全来自**并行度**：并行模型同时执行 4 任务，串行模型依次执行
3. 单线程协作式模型的总耗时 ≈ concurrency × 单任务延迟
4. tokio 内存最省（4.0MB）——无独立协程栈，线程池复用

concurrency=1 的单任务延迟基线：

| Runner | tokio | thread | process | green | stackless | embassy |
|--------|:---:|:---:|:---:|:---:|:---:|:---:|
| checksum-light | 70.2ms | 69.8ms | 71.1ms | 68.3ms | 67.7ms | 68.3ms |

> **所有模型的单任务延迟在测量误差范围内完全一致**，确认计算本身无差异，差异仅来自并行调度。

### 硬件并发 vs 软件协作

```
多核并行 (tokio/thread/process):     单线程协作 (green/stackless/embassy):
┌────┬────┬────┬────┐                ┌────────────────────────────┐
│ T1 │ T2 │ T3 │ T4 │                │ T1→T2→T3→T4                │
│    │    │    │    │                │                            │
│ 4核同时运行 4任务   │                │ 1核轮流运行 4任务           │
│ 总时间 ≈ 73ms       │                │ 总时间 ≈ 283ms             │
│ (实测 thread)       │                │ (实测 green-thread)        │
└────────────────────┘                └────────────────────────────┘
加速比 = 283/73 ≈ 3.9× ≈ concurrency=4
```

这是协作式调度的基本物理限制，也是为什么 tokio 对 I/O 密集场景优秀而对 CPU 密集场景需要 `spawn_blocking`。

## 四、优先级调度深度分析

### 调度策略矩阵

| 策略 | 机制 | 公平性 | 饿死风险 | 实现复杂度 |
|------|------|:---:|:---:|:---:|
| **CFQ** (vruntime) | vruntime += time × 1024/weight | ★★★★★ | 无 | ~200 行 |
| **HighestPrioFirst** | find_highest_priority_ready() | ★★☆☆☆ | 有 | ~20 行 |
| **RoundRobin** | 循环轮转 | ★★★★★ | 无 | ~15 行 |
| **Tokio 三队列** | 按优先级批量分发 | ★★★☆☆ | 低 | ~120 行 |
| **Embassy 抢占式** | 中断抢占 + bitmap | ★★★★☆ | 无 | 框架内置 |

### 满载 CPU 下优先级效果（来自 References 实验数据）

33 所高校，满载 CPU (cpu_repeat=10000)，三批优先级（PRIO=2/1/0）：

| 调度器 | 总均延迟 | 高优批延迟 | 低优批延迟 | 差异化倍数 |
|--------|:---:|:---:|:---:|:---:|
| GT Priority | 910.4ms | 1123.7ms | 327.7ms | **3.4×** |
| GT RoundRobin | 1522.1ms | 1417.4ms | 1585.9ms | 1.1× |
| Tokio Priority | 727.4ms | — | — | — |
| Tokio Default | 726.3ms | — | — | 1.0× |

> - 绿色线程的 `HighestPriorityFirst` 在满载 CPU 下实现了 3.4× 的优先级延迟差异
> - 协作式调度下优先级生效依赖任务的主动 yield，无 yield 时完全串行化
> - Tokio 无原生优先级支持，三队列分发器经优化后 Priority vs Default 差距仅 2.4%

### Tokio 分发器优化（BinaryHeap → 三队列）

| 优化阶段 | 分发次数 | Priority 耗时 | 与 Default 差距 |
|---------|:---:|:---:|:---:|
| 优化前 (BinaryHeap) | 33 次 | 6.35s | +1.83s (**40%**) |
| 优化后 (三队列 Vec) | 3 次 | 4.55s | +0.11s (**2.4%**) |

**教训**：
1. O(log n) 不是瓶颈（33×O(log 33) ≈ 50ns，占总耗时 0.0008%）
2. 真正的瓶颈在**串行分发循环**——33 次 spawn 形成单点瓶颈
3. user time vs wall time 差异是诊断关键线索（user time 相同而 wall time 不同 → 问题在调度/分发而非计算）
4. 使用框架原语（Semaphore）比 DIY（AtomicUsize+Notify）更高效

## 五、Embassy / Embassy-Preempt 探索 (Task 3)

### 架构特点

Embassy 是为嵌入式设计的抢占式异步运行时：

- **无需一直轮询**：无任务时 CPU 通过 WFE/SEV 进入休眠
- **多执行器实例**：不同优先级各有一个执行器，中断可抢占低优先级任务
- **零堆分配**：任务在编译期通过 `#[embassy_executor::task]` 定义
- **三层执行器**：thread → interrupt → preempt（信号量实现优先级）

### Task 3 贡献

为 `embassy-preempt` 实现了两个关键扩展：

**1. 同优先级 Round-Robin**（改动约 100 行）：

```
单任务 per 优先级               循环双向链表
┌──────────┐                   ┌──────────────────────┐
│ os_prio  │                   │ Head → Task1 → Task2 │
│ _tbl[P]  │                   │   ↑                 ↓ │
│   → T1   │                   │   └──── Task3 ←─────┘ │
└──────────┘                   └──────────────────────┘
新增 OSTCBRdyNext/Prev 字段，支持多任务共享同一优先级
```

**2. Mutex (优先级天花板协议)**（新增约 220 行）：


## 六、my_mini_redis — 基于 Tokio 的 Redis 兼容内存数据库

### 架构设计

```
┌──────────────────────────────────────────────────────────┐
│                   my_mini_redis                          │
├──────────────────────────────────────────────────────────┤
│  TCP Server (tokio::net::TcpListener)                    │

│    ├─ 16 shards, key → hash → shard_index                │

│    └─ Lazy TTL expiry on access                          │
├──────────────────────────────────────────────────────────┤
│  Data Types                                              │
│    ├─ String: Vec<u8>                                    │
│    ├─ List:   VecDeque<Vec<u8>>                          │

│    └─ Set:    HashSet<Vec<u8>>                           │
└──────────────────────────────────────────────────────────┘
```

### 分片并发存储

```rust
pub const NUM_SHARDS: usize = 16;
pub type Shard = Mutex<HashMap<String, DbEntry>>;
pub type ShardedDb = Arc<Vec<Shard>>;

fn shard_index(key: &str) -> usize {
    let mut h = DefaultHasher::new();
    key.hash(&mut h);
    (h.finish() as usize) % NUM_SHARDS
}
```

16 个分片各自独立加锁，不同 key 的并发操作不会互相阻塞。TTL 过期采用惰性删除：访问时判断 `Instant::now()` 与 `expires_at`，过期则自动移除。

| 层级 | 命令 | 说明 |
|------|------|------|
| **T1 Basic** | PING, SET, GET, DEL, EXISTS, KEYS, FLUSHALL, DBSIZE | 基础 KV 操作 |

### TCP 服务器

```rust
#[tokio::main]
async fn main() {
    let listener = TcpListener::bind("127.0.0.1:6379").await.unwrap();
    let db = new_db();
    loop {
        let (socket, addr) = listener.accept().await.unwrap();
        let db = db.clone();
        tokio::spawn(async move {
            let mut connection = Connection::new(socket);
            while let Some(frame) = connection.read_frame().await.unwrap() {
                let response = dispatch(&db, frame);
                connection.write_frame(&response).await.unwrap();
            }
        });
    }
}
```

每个客户端连接由一个独立的 tokio task 处理。多个客户端可并发读写，分片锁粒度保证高并发下的低竞争。

### MiniTokio — 85 行迷你异步运行时

```rust
struct MiniTokio {
    tasks: VecDeque<Task>,  // Pin<Box<dyn Future<Output = ()> + Send>>
}

impl MiniTokio {
    fn run(&mut self) {
        let waker = task::noop_waker();
        let mut cx = Context::from_waker(&waker);
        while let Some(mut task) = self.tasks.pop_front() {
            if task.as_mut().poll(&mut cx).is_pending() {
                self.tasks.push_back(task);  // 放回队尾等待
            }
        }
    }
}
```

核心逻辑仅 10 行：任务队列 + FIFO 轮询 + Pending 放回。配合 `Delay` Future（线程睡眠 + wake 通知），完整演示了 async 运行时的本质：

- **Executor**（MiniTokio）负责 poll Future
- **Reactor**（Delay 内部线程）负责异步等待外部事件并调用 waker.wake()
- **Waker** 连接二者：Reactor 通知 Executor 重新 poll

### 选型建议

| 场景 | 推荐运行时 | 原因 |
|------|-----------|------|
| I/O 密集型高并发服务 | **tokio** | epoll + work-stealing + 生态成熟 |
| CPU 密集型并行计算 | **std::thread** / rayon | OS 级抢占 + 无调度层开销 |
| 嵌入式实时控制 | **embassy-preempt** | 中断抢占 + bitmap 优先级查找 |
| 教学/理解调度原理 | **green-thread (CFQ)** | 代码量小（~360行），概念清晰 |
| 资源敏感 (内存 < 1MB) | **embassy** | 零堆分配，编译期任务定义 |
| 混合 I/O + CPU | **tokio + spawn_blocking** | I/O 走 async，CPU 走线程池 |

## 八、内存开销显微分析

| 组件 | 大小 | 占绿色线程总内存 |
|------|------|:---:|
| Thread 结构体 | 112 B | 0.005% |
| ThreadContext (7 regs) | 56 B | 0.003% |
| priority 字段 | **1 B** | **0.000048%** |
| 协程栈 (2MB) | 2,097,152 B | **99.992%** |

> `priority` 字段嵌入已有 8 字节对齐 padding，不改变 `size_of::<Thread>()`。调度算法均为 O(n) 线性扫描，不产生额外堆分配。

| 运行时 | 峰值 RSS | 分析 |
|--------|:---:|------|
| tokio | **4.0 MB** | 线程池复用，无独立协程栈 |
| thread | 7.8 MB | 4 个 OS 线程各 ~2MB 实际 RSS |
| process | 7.9 MB | CoW 父子进程，与 thread 接近 |
| green-thread | 8.9 MB | 4 协程各 2MB 栈 + Runtime |
| stackless | 8.9 MB | Future 状态机 on heap |
| embassy | 8.9 MB | 同单线程进程基线 |

> 数据来自 concurrency=4, checksum-light 实测。tokio 内存最省（无额外栈分配）。

## 九、总结与展望

### 核心收获

1. **异步 ≠ 并行**：async/await 解决的是 I/O 等待的效率问题，而非 CPU 计算的加速问题。纯 CPU 任务在单线程协作式模型上的吞吐率恰好是 CPU 核数分之一。

2. **CFQ 公平调度在所有模型中均验证通过**：等权重公平轮转、高权重优先、低优先级不饿死——58 个测试全部通过。

3. **优先级调度有效但依赖 yield 注入**：协作式调度下，优先级效果来自任务的主动 yield。无 yield 时任务完全串行化。

4. **调度器开销的来源不在"选择"而在"分发"**：BinaryHeap 的 O(log n) 不是瓶颈（33 次仅 50ns），真正的瓶颈在架构层的串行分发循环。

5. **Embassy 为嵌入式异步提供了优雅的解决方案**：中断抢占 + bitmap 优先级查找 + WFE 休眠，在资源极度受限的场景下实现了高效的异步调度。

### 展望

1. **更多运行时对比**：可扩展对比 monoio (io_uring)、glommio (共享无) 等
2. **能效比分析**：嵌入式场景下加入功耗数据
3. **实时性指标**：增加 deadline miss rate 分析
4. **Embassy 真实硬件测试**：在星光 2 开发板上跑 embassy-preempt benchmark
